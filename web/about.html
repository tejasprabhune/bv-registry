the problem

Bioinformatics pipelines fail in predictable ways. Conda environments drift between machines. Pinning a version string (samtools==1.20) means something different three months later when its dependencies change. Docker fixes the runtime environment, but most workflows record only a tag in their config—and tags are mutable.

The result: analyses that ran fine in January produce different output in March, or don't run at all on the HPC cluster because Docker isn't permitted there.

what bv does

bv records the sha256 manifest digest of every tool in bv.lock. That hash covers the exact image bytes pulled and run—not a tag, not a version string. Commit the lockfile; on any other machine, bv sync reproduces that state by digest or fails loudly.

bv add <tool> writes bv.toml and bv.lock and generates PATH shims under .bv/bin/. Existing scripts that call blastn or samtools by name work without modification—no wrappers, no activation step.

The same lockfile works on Docker (typical on laptops) and Apptainer (typical on HPC clusters where Docker is not available). Same image, same digest, different runtime.

install

curl -fsSL https://raw.githubusercontent.com/tejasprabhune/bv/main/install.sh | sh
# or: cargo install biov

Runs on Linux and macOS. Windows: use WSL2 with Docker Desktop's WSL2 backend; native Windows is not yet supported.

tl;dr

bv add blast
bv exec blastn -query foo.fa -db nr
git add bv.toml bv.lock     # commit the pin; `bv sync` reproduces anywhere

container runtime

bv shells out to Docker or Apptainer—install one first: 

# Docker, typical on a laptop
curl -fsSL https://get.docker.com/rootless | sh

# Apptainer, typical on HPC nodes (no daemon, no root required)
conda install -c conda-forge apptainer

Docker is the simpler choice on a laptop. Apptainer runs without a daemon and doesn't need root—which is why most HPC clusters allow it where Docker is not. For GPU tools on Docker, install nvidia-container-toolkit. Apptainer uses the host's NVIDIA libraries directly with --nv.
